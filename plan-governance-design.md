# plan-governance 设计文档

## 1. 背景

`plan-governance` 是一种面向个人或小团队的轻量文档治理机制。它不是普通任务清单，也不是大型开源社区的正式提案审批流程。它解决的问题是：当一个任务无法一次完成、涉及多个阶段、会影响公共契约，或会和已有计划及决策发生关系时，如何避免计划漂移、旧决策被覆盖、阶段边界失控和完成证据缺失。

该机制的核心原则是：

- 普通任务直接实施，不引入治理流程。
- 只有任务具有跨时间状态时才启用治理。
- 当前阶段写细，后续阶段写粗。
- 决策、计划、契约和证据各自有权威位置，不在多个文档中重复定义。
- 计划文档服务实施，不替代实施。

## 2. 设计目标

`plan-governance` 的目标是为持续演进任务提供最小但足够的治理结构：

- 识别哪些任务需要持续治理，哪些任务不需要。
- 把长期任务拆成可交付、可验证的阶段。
- 维护计划之间的依赖、替代、重叠和推荐顺序。
- 显式管理阻塞问题和延后问题。
- 将关键决策沉淀到 ADR，将精确行为沉淀到 spec 或 Schema。
- 在每个阶段保留可验证的完成证据。
- 通过脚本或 CI 做基础一致性检查。

## 3. 非目标

`plan-governance` 不试图成为完整项目管理系统：

- 不替代 issue tracker、看板或 release 管理。
- 不要求每个任务都写计划文档。
- 不要求每个计划都配 ADR。
- 不强制模拟大型开源社区的正式审批流程。
- 不用文档流程阻塞小修小改。
- 不在早期阶段预写所有未来阶段的详细步骤。

## 4. 适用范围

启用治理的典型信号：

- 跨阶段或里程碑实施。
- 多个计划之间存在依赖、替代、重叠或冲突。
- 涉及架构、公共接口、Schema 或兼容性迁移。
- 当前任务无法一次完成。
- 后续需要真实运行报告或样本反馈。
- 同一模块会被多个后续计划反复修改。
- 用户明确提出路线图、持续优化或逐步迁移。

判定规则：

| 分数 | 处理方式 |
|---|---|
| `>=3` | 启用持续计划治理 |
| `=2` | 检查已有治理文档，但不初始化新结构 |
| `<=1` | 按普通任务处理 |

大型公共契约迁移即使缺少其他信号，也应直接启用治理。互不依赖的待办列表不视为持续计划。

## 5. 开源方案借鉴

`plan-governance` 不是直接复制某一个开源规范，而是借鉴多种成熟模式后收缩成个人开发可承受的版本。

### 5.1 ADR / MADR

借鉴点：

- 用独立文档记录关键架构决策。
- 每个 ADR 说明背景、决策、替代方案和后果。
- 已经确认的决策不只停留在对话或计划正文里。

在 `plan-governance` 中的映射：

- `docs/adr/` 存放关键决策。
- 计划文档只引用 ADR，不复制决策全文。
- 当计划方向变化违背既有 ADR 时，必须先更新或新增 ADR。

不照搬的部分：

- 不要求所有实现选择都写 ADR。
- 不把 ADR 当作任务计划或实施步骤。

参考：

- https://github.com/adr/madr
- https://adr.github.io/

### 5.2 RFC Process

借鉴点：

- 未定案的设计先以提案形式讨论。
- 记录动机、方案、未决问题和接受条件。
- 适合跨模块、跨团队或公共语义变化。

在 `plan-governance` 中的映射：

- 未决问题进入 `PLAN_MAP` 或当前计划的 pending queue。
- 阻塞当前阶段的问题优先确认。
- 非阻塞问题明确延后到后续阶段。

不照搬的部分：

- 不要求每个计划都走正式 RFC 生命周期。
- 不引入复杂的社区投票或审批状态。

参考：

- https://github.com/rust-lang/rfcs
- https://github.com/reactjs/rfcs

### 5.3 Kubernetes KEP

借鉴点：

- 用 enhancement proposal 管理长期演进。
- 明确状态、阶段、风险、验收和发布影响。
- 适合跨阶段特性、迁移和公共契约演进。

在 `plan-governance` 中的映射：

- `docs/plans/<plan>.md` 类似轻量 KEP。
- 当前阶段必须有进入条件、实施步骤和完成证据。
- 后续阶段只保留目标、依赖、风险和验收方向。

不照搬的部分：

- 不要求 SIG、release team、正式 graduation 阶段。
- 不强制维护大型社区所需的完整元数据。

参考：

- https://github.com/kubernetes/enhancements

### 5.4 OpenStack Specs

借鉴点：

- 用规格文档表达问题、方案、替代方案、影响面和测试。
- 按版本或状态组织设计规格。
- 适合评审一个设计是否应该被接受。

在 `plan-governance` 中的映射：

- 单个 `docs/plans/<plan>.md` 可借鉴 OpenStack spec 的结构。
- 对公共 API、数据模型、安全、兼容性的影响需要显式写出。
- 已完成计划需要保留验证证据。

不照搬的部分：

- 不采用完整的 release-driven `approved/implemented` 目录流转。
- 不绑定 Blueprint、Launchpad、Gerrit 或大型社区评审机制。
- `PLAN_MAP.md` 不等于 OpenStack spec；它更像计划索引、依赖图和风险面板。

参考：

- https://github.com/openstack/neutron-specs

### 5.5 Docs-as-code / Backstage TechDocs

借鉴点：

- 文档与代码同仓库维护。
- 文档变更进入 review 和 CI。
- 通过统一入口提高发现性。

在 `plan-governance` 中的映射：

- 治理文档存放在 `docs/` 下。
- 计划、ADR、迁移说明和验证证据都可被版本控制追踪。
- 可结合 Markdown lint、链接检查、术语检查和自定义一致性脚本。

不照搬的部分：

- 不要求搭建完整文档门户。
- 不要求所有项目都引入 Backstage。

参考：

- https://backstage.io/docs/features/techdocs/

## 6. 核心文档模型

推荐结构：

```text
docs/
  PLAN_MAP.md
  plans/
    <plan-name>.md
  adr/
    0001-<decision>.md
  migrations/
    <migration-name>.md
```

各文档的权责：

| 文档 | 权威内容 |
|---|---|
| `docs/PLAN_MAP.md` | 计划索引、状态、依赖、替代关系、推荐顺序、阻塞问题 |
| `docs/plans/<plan>.md` | 目标、范围、阶段、当前阶段步骤、字段方案、Schema、枚举、Step 0 证据、验证方式、完成条件、剩余风险 |
| `docs/adr/*.md` | 决策原因、被选方案、替代方案、后果 |
| `docs/migrations/*.md` | 兼容策略、迁移步骤、回滚方式、旧行为保留窗口 |
| spec / Schema | 精确行为、字段定义、机器可校验契约 |

同一事实只定义一次。其他文档通过链接引用，避免复制后产生漂移。

事实源优先级：

- 专项计划是实施细节事实源，记录字段方案、Schema、枚举、Step 0 证据、验证方式和完成条件。
- `PLAN_MAP.md` 是状态、依赖、替代/合并/废弃关系、推荐顺序、阻塞项和证据链接的事实源。
- 总路线图、优先级计划和索引只记录顺序、状态摘要和专项计划链接，不复制字段级方案、枚举、Step 0 细节或完成定义。
- spec 或 Schema 是可机器校验契约的事实源；计划文档只链接它们或说明当前阶段如何迁移。
- 启用治理后，已有草案、历史设计、归档计划、临时分析文档等默认只作为背景材料，不再作为规范事实源。

## 7. PLAN_MAP 设计

`PLAN_MAP.md` 是治理入口，不是所有计划细节的汇总。

建议包含：

- 当前治理范围。
- 计划索引。
- 计划状态。
- 计划依赖。
- 替代、合并、废弃关系。
- 推荐实施顺序。
- 阻塞当前阶段的未决问题。
- 已完成阶段的验证证据链接。

不建议包含：

- 字段级 schema、枚举或接口定义。
- 专项计划的 Step 0 细节、完整实施步骤或完成定义。
- 可以从专项计划、ADR、迁移文档或 spec 链接取得的重复事实。

计划状态建议：

| 状态 | 含义 |
|---|---|
| `候选` | 记录了想法，但尚未承诺实施 |
| `设计中` | 正在明确范围、契约和门禁 |
| `待实施` | 当前阶段门禁已通过，但尚未开始 |
| `实施中` | 当前阶段正在修改代码或文档 |
| `已完成` | 实现、测试、证据和文档已同步 |
| `已替代` | 被另一个计划取代 |
| `已合并` | 并入另一个计划 |
| `已废弃` | 明确不再推进 |

未决问题表：

| 问题 | 推荐方案 | 影响范围 | 是否阻塞当前阶段 | 状态 |
|---|---|---|---|---|
| 示例问题 | 示例推荐方案 | 示例影响范围 | 是 / 否 | 待确认 / 已决定 / 延后到阶段 N |

## 8. Plan 文档设计

单个计划文档建议包含：

- 背景。
- 目标。
- 非目标。
- 不变量。
- 影响模块或文件。
- 公共契约变化。
- 阶段路线图。
- 当前阶段实施步骤。
- Step 0 证据。
- 验证方式。
- 完成条件。
- 未决问题。
- 风险和回滚。
- 关联 ADR、migration、spec 或 issue。

阶段细化规则：

- 当前阶段写到可执行步骤。
- 下一阶段写进入条件、主要风险和验收方向。
- 更晚阶段只写目标、依赖、非目标和验收方向。

不要在总计划中复制所有阶段步骤，也不要在阶段计划中复制 Schema 字段定义。

## 9. Step 0 证据

每个当前阶段都需要先固定证据，再改实现。

| 任务类型 | Step 0 |
|---|---|
| 缺陷修复 | 失败回归测试或最小复现 |
| 行为迁移 | 现状快照、兼容样本或契约测试 |
| 新项目 | 可执行验收 fixture、提交图或协议样本 |
| 架构探索 | 最小原型、风险实验或关键假设验证 |

Step 0 的作用是防止计划只有意图，没有可观察基线。

## 10. 阶段门禁

当前阶段进入实施前，需要满足：

- 当前阶段目标、范围和非目标明确。
- 角色、信任边界和不变量明确。
- 当前阶段依赖的公共契约及失败策略明确。
- Step 0 和验收方式可执行。
- 未决队列中没有阻塞当前阶段的问题。

后续阶段仍有未决问题不构成当前阶段阻塞。

## 11. 漂移检查

在重要决策确认后、实施前、收到新运行报告后，都需要回答：

- 新信息是否违背既有 ADR 或核心原则？
- 是否改变其他计划的前置条件、顺序或有效性？
- 是否改变公共接口、Schema、状态语义或兼容承诺？
- 是否出现计划外的重复修改点？
- 原有测试和完成条件是否仍能证明目标达成？
- 专项计划的状态、字段方案、完成条件或验证结果是否需要同步到 `PLAN_MAP` 和引用它的路线图、优先级计划或索引？

如果答案为“是”，先更新 `PLAN_MAP` 和相关计划，再继续实施。

验收治理文档时，必须做反向引用检查：

- 用 `rg` 搜索同名计划、P 编号、状态名和关键字段。
- 检查所有引用该计划或事项的路线图、优先级计划、索引和专项计划。
- 如果同一事实在多个文档中重复，保留一个事实源，其他文档改为链接引用。
- 搜索并修正 `草案为准|以草案为事实源|详见草案|draft is source|source of truth.*draft|以.*draft.*为准` 等旧草案重新成为事实源的表达。

## 12. 草案和历史文档处理

启用计划治理前，草案、README、临时分析或历史设计文档可以作为讨论载体。启用治理后，事实源必须切换到治理文档和正式 spec。

- 旧草案、历史设计、归档计划和临时分析文档默认只作为背景材料。
- 除非用户明确点名要求更新这些文档，否则不要继续修改它们来承载新规范。
- 新发生的目标、范围、非目标、公共契约、字段、Schema、状态语义、阶段、验证方式、完成条件、风险和回滚，应写入 `docs/plans/*.md`、ADR、migration 或正式 spec。
- 计划状态、依赖、替代/合并/废弃关系、推荐顺序、阻塞项和证据链接，应写入或同步 `docs/PLAN_MAP.md`。
- 当用户说“补充一下”“记录一下”“把刚才的结论写进去”时，如果治理已启用，默认写入治理文档，不写入草案、README 或历史设计文档，除非用户明确点名该文件。
- 历史草案可以在治理文档中作为背景链接引用，但不应再承载新的规范事实。

## 13. 自动化检查

治理文档采用 `docs/PLAN_MAP.md` 和 `docs/plans/` 结构时，应提供基础一致性检查脚本。

最低检查项：

- 所有被 `PLAN_MAP` 引用的计划文件存在。
- 计划状态值合法。
- 计划依赖不存在环。
- 已完成计划有验证证据。
- 已替代或已废弃计划没有被实施中计划依赖。
- 实施中的计划没有阻塞当前阶段的未决问题。
- 文档以 UTF-8 读取无乱码。

可选检查项：

- Markdown lint。
- 链接检查。
- Vale 术语和风格检查。
- Spectral 检查 OpenAPI、AsyncAPI、JSON Schema 或 YAML 契约。

## 14. 个人开发取舍

个人开发最容易过度设计，因此默认策略应保持克制：

| 场景 | 建议 |
|---|---|
| 一次性 bugfix | 不建治理文档 |
| 小功能 | issue 或简短 checklist |
| 两三步小重构 | 可写单个 `docs/plans/<task>.md` |
| 跨阶段迁移 | 使用 `PLAN_MAP.md + plan doc` |
| 公共接口或 Schema 变化 | 增加 migration 或 spec |
| 不可逆架构选择 | 增加 ADR |
| 需要长期反馈调整 | 在计划中保留证据和风险队列 |

推荐规则：

> 普通任务直接做；只有跨阶段、影响公共契约、依赖真实反馈，或会和旧计划冲突时，才启用 `PLAN_MAP + plan + ADR`。

## 15. 与 OpenStack Specs 的区别

OpenStack specs 更适合大型开源社区的设计审批。它关注某个设计是否应进入某个 release，并依赖社区评审、状态目录和外部任务系统。

`plan-governance` 更适合个人或小团队的持续实施治理。它关注多个计划和阶段如何一致推进，如何管理阻塞问题，如何保留验证证据，以及如何避免新计划覆盖旧决策。

| 维度 | OpenStack Specs | plan-governance |
|---|---|---|
| 核心对象 | 单个功能或变更的设计规格 | 多个计划、阶段、决策、迁移和证据 |
| 主要问题 | 设计是否被接受 | 计划是否一致、可执行、可验证 |
| 生命周期 | proposed / approved / implemented | 候选 / 设计中 / 待实施 / 实施中 / 已完成 / 已替代 |
| 组织方式 | 按 release 和状态目录组织 | 以 `PLAN_MAP` 为入口，链接 plans、ADR、migrations |
| 决策记录 | 常在 spec 内描述 | 关键决策进入 ADR |
| 实现跟踪 | 依赖外部系统 | 计划文档记录阶段状态和证据 |
| 适合规模 | 大型社区 | 个人或小团队 |

## 16. 最小落地方案

如果一个项目第一次启用治理，建议只创建：

```text
docs/
  PLAN_MAP.md
  plans/
    current-plan.md
```

只有在实际发生关键架构决策时创建：

```text
docs/
  adr/
    0001-example-decision.md
```

只有在实际发生公共契约迁移时创建：

```text
docs/
  migrations/
    example-migration.md
```

不要预建空目录，也不要为了“规范完整”创建没有内容的治理文件。

## 17. 成功标准

`plan-governance` 成功不是因为文档很多，而是因为：

- 当前该做什么一眼可见。
- 阻塞项没有藏在对话里。
- 已经做出的关键决策能被追溯。
- 公共契约变化有迁移和回滚说明。
- 已完成阶段有测试、fixture、样本或运行报告作为证据。
- 后续计划不会静默覆盖旧计划。
- 普通任务没有被流程拖慢。
