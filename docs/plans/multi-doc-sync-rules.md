# 计划：multi-doc-sync-rules

## 背景

现有 `plan-governance` 规则已经要求检查 `docs/PLAN_MAP.md`、避免同一事实重复定义，并在漂移时更新相关计划。但反馈指出，多文档同步、总路线图与专项计划的边界、验收时的反向引用检查，以及事实源优先级还不够明确。

## 目标

补强 `plan-governance` 规则，使路线图、优先级计划、索引、`PLAN_MAP.md` 和专项计划之间的权责边界清晰，并要求验收治理文档时检查反向引用和重复事实。

## 非目标

- 不重写自动化检查脚本为完整语义漂移检测器。
- 不引入新的文档目录结构。
- 不修改既有计划状态枚举。

## 不变量

- 同一事实只定义一次，其他文档通过链接引用。
- 当前阶段写细，后续阶段写粗。
- 生成到项目的执行规则必须和 skill 源规则保持一致。

## 影响模块或文件

- `/Users/jafish/.codex/skills/plan-governance/SKILL.md`
- `README.md`
- `plan-governance-design.md`
- `scripts/init_plan_governance.py`
- `tests/test_init_plan_governance.py`
- `docs/PLAN_MAP.md`

## 公共契约变化

治理文档约定新增多文档同步规则：

- 专项计划是实施细节事实源。
- `PLAN_MAP.md` 是状态、依赖、替代/合并/废弃关系、推荐顺序、阻塞项和证据链接的事实源。
- 总路线图、优先级计划和索引只记录顺序、状态摘要和专项计划链接。
- 专项计划变化时，必须同步所有引用该计划的路线图、优先级计划或索引。
- 验收治理文档时，必须用 `rg` 搜索同名计划、P 编号、状态名和关键字段。

### Step 0 证据

反馈指出现有规则没有足够明确约束以下场景：

- 总路线图复制专项计划字段方案，导致专项计划确认字段后总路线图漂移。
- 专项计划状态变化后，引用它的路线图或索引未同步。
- 验收治理文档时只看单个文件，没有搜索同名计划、状态、P 编号和关键字段。
- 专项计划、`PLAN_MAP.md`、总路线图之间的事实源优先级不够明确。

### 验证方式

- 运行 `python3 -m pytest`。
- 运行 `python3 scripts/check_plan_governance.py .`。
- 用 `rg` 搜索 `多文档同步|事实源|不复制字段级方案|rg.*同名计划`，确认规则进入 skill、README、设计文档、初始化脚本和 `PLAN_MAP.md`。

### 测试覆盖率

`python3 -m pytest` 通过，测试覆盖率 98.54%，高于 85% 门禁。

### 完成条件

- 生成到项目的 `CLAUDE.md` 包含事实源规则和反向引用验收规则。
- README、设计文档和本仓库 `PLAN_MAP.md` 同步描述同一事实源层级。
- 测试和治理检查通过。

## 风险和回滚

风险：规则过宽导致普通小任务也被要求做多文档同步。

控制：保留启用判定和普通任务跳过治理的规则；多文档同步只约束已经进入治理范围的文档。

回滚：恢复 `SKILL.md`、文档和初始化脚本中新增的多文档同步规则，并删除本计划在 `PLAN_MAP.md` 中的引用。
